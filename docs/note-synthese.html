Architecture choisie

J'ai choisi l'Option B — système agentique. Ce qui m'a motivé dans ce choix, c'est d'abord mon intérêt grandissant pour les agents et les systèmes agentiques, et plus anciennement pour les architectures structurées. Au-delà de ça, cette approche offre plusieurs avantages concrets par rapport à une automatisation type Make :

  • Moins déterministe — prend mieux en compte les contextes et les cas ambigus
  • Plus scalable — les agents s'ajoutent sans refondre le pipeline
  • Plus résilient — les cas imprévus sont gérés dans la logique, pas contournés
  • Plus facile à modifier — prompts externalisés en fichiers .md, swappables sans redéploiement

Où se loge la compliance

La conformité est dans le design du scorer, pas en post-traitement. Elle bloque avant que le score OPAD soit calculé — pas après.

Gate bloquant
RGPD, IA Act, PI — si l'un de ces trois scores ≤ 3, le contenu est rejeté immédiatement, sans calcul du score OPAD. La raison : un contenu non conforme ne doit jamais entrer dans le catalogue, quelle que soit sa qualité pédagogique.
Revue humaine
Inclusion (a_inclusion ≤ 3) — déclenche une revue humaine obligatoire. Le LLM extrait le passage exact problématique (flag_excerpt). Un humain valide avant toute publication.
PI dédiée
Routes /api/pi-sources et /api/pi-validate séparées pour la vérification propriété intellectuelle sur les sources — découplées du pipeline principal, activables indépendamment.

Ce que le système ne peut pas faire seul

La qualification automatique complète est impossible par design — pas un bug, une contrainte réglementaire. Deux gates ne peuvent pas être délégués à un LLM :

IA Act
Tout contenu potentiellement généré par IA doit être signalé et validé par un humain avant diffusion en contexte de formation professionnelle. Le LLM détecte le signal — il ne peut pas certifier la transparence au sens réglementaire.
RGPD
Un modèle de langage peut détecter des patterns de données personnelles — il ne peut pas certifier la conformité. Cette certification reste une responsabilité humaine et juridique.

Ce que le système fait réellement : passer de ~45 min à ~3 min de revue par contenu. Il trie, détecte, documente, justifie. La revue humaine reste — mais ciblée sur les 20 à 30 % de cas ambigus, pas sur la totalité du flux.

3 points de rupture techniques à anticiper

1
Le polling n8n séquentiel devient un goulot
À 143 contenus/jour, le poll toutes les 2 min ne suit plus en séquentiel. Solution : file Redis (LPUSH/BRPOP) avec workers parallèles — Notion alimente la queue, plusieurs instances /api/intake consomment en parallèle. Notion Business déverrouille les rate limits nécessaires.
2
Les APIs d'extraction sont des points de défaillance uniques
Transcript.com et YouTube Data API ont des pannes et des quotas. À volume, une source indisponible bloque tout le pipeline. Solution : retry exponentiel (3 tentatives) + fallback Whisper via Groq pour les podcasts + cache Redis 24h des transcripts déjà récupérés.
3
Redis seul ne suffit pas pour l'historique de qualification
Les résultats de qualification sont précieux à volume (audit, re-scoring, eval set). Redis avec TTL n'est pas une base historique. Solution : migrer les résultats long-terme vers PostgreSQL (Neon ou Supabase), Redis reste pour les verrous, sessions et cache chaud. Coût additionnel : ~10-15 €/mois.
